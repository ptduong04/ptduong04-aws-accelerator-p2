Vấn đề thực tế

Deploy tay từ local

kubectl apply thì dễ. Vận hành thật thì sập. Tại sao?

1Không ghi lại state

Developer kubectl từ laptop → không ai biết cụm đang chạy gì.

Bạn deploy v2.5 lúc 3h chiều. 3 ngày sau sếp hỏi: "Prod chạy version nào?" — không nhớ.

2Nhầm context

Lệnh đúng nhưng sai cụm → xóa nhầm prod.

Junior nghĩ đang ở DEV, gõ kubectl delete deployment. Nhìn lại: context = PROD.

3Không rollback được

"Về như hôm qua" — nhưng "hôm qua" là cấu hình nào? Không có ảnh chụp để quay về.

kubectl rollout undo chỉ lưu 10 revisions, không có message. History chỉ có số, không biết "version này làm gì".

4Credential cũ còn

Người nghỉ việc vẫn giữ kubeconfig → rủi ro.

Bạn A nghỉ. File kubeconfig của A vẫn hoạt động → vô tình hoặc cố ý vẫn apply được vào cụm.

Gốc rễ: Với deploy tay, không có "single source of truth" — không biết cụm đang chạy gì, ai đổi, đổi lúc nào.

Hiểu về concept của GitOps

Quản lý file cấu hình (manifest) trên Git.
Agent của ArgoCD hỗ trợ việc tự động triển khai applications.
Không kubectl tay

Khái niệm

4 nguyên tắc OpenGitOps

Git = nguồn sự thật duy nhất (single source of truth)

Nguyên tắcGiải thíchVí dụ minh họa
1. Declarative Khai báo "muốn gì" (desired state), không viết các bước "làm thế nào" Viết replicas: 3 (muốn 3 pods)
Thay vì viết script: "tạo pod 1, đợi ready, tạo pod 2..."
2. Versioned Mọi thay đổi lưu trong Git, có lịch sử đầy đủ git log xem ai đổi, lúc nào, đổi gì.
Rollback = git revert (có audit trail)
3. Pulled Agent trong cụm tự kéo từ Git ra. Không ai push từ ngoài vào ArgoCD (chạy trong cụm) poll repo mỗi 3 phút.
Developer không cần kubeconfig của prod → an toàn hơn
4. Reconciled Liên tục so sánh Git vs cụm thật. Nếu lệch → tự động sửa về như Git Git: replicas: 3. Bạn kubectl scale 5 → 5 pods.
ArgoCD phát hiện: "Git nói 3 mà!" → tự scale về 3.
Giống ReplicaSet reconcile, nhưng source = Git

Model: Git = bản thiết kế; ArgoCD = thợ sửa cụm cho khớp. Reconcile loop giống HPA, nhưng desired từ Git.

End to End

Điều gì xảy ra khi bạn git push một thay đổi?

# 1. Bạn sửa replicas 2 -> 4, rồi commit + push lên GitHub git commit -am "frontend 2->4" && git push ↓ # 2. ArgoCD (trong cụm) poll repo / nhận webhook -> thấy commit mới # 3. repo-server kéo manifest mới về và render # 4. application-controller so: Git nói 4, cụm đang 2 -> OutOfSync # 5. auto-sync: apply Deployment(replicas=4) vào namespace demo # 6. K8s tạo thêm 2 Pod -> 4 Pod. ArgoCD -> Synced / Healthy

Khác biệt: Không chạm cụm. Chỉ chạm Git. ArgoCD làm còn lại. Có độ trễ (vài phút) — trade-off của Pull. Đổi lại: an toàn + tự động.

Đối tượng cốt lõi

Application — "chỉ dẫn" cho ArgoCD

CRD ở ns argocd, trỏ tới app.

apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: platform namespace: argocd spec: source: # Ở ĐÂU? repoURL: https://github.com/<ban>/gitops.git targetRevision: main # branch nào path: k8s # thư mục nào destination: # Apply vào ĐÂU? namespace: demo server: https://kubernetes.default.svc syncPolicy: # Tự động sync? automated: prune: true # xóa nếu xóa khỏi Git selfHeal: true # tự sửa nếu ai đó edit tay

Đọc trường

  • source = Desired ở đâu (repo + branch + path)
  • destination = Apply vào đâu (ns + cluster)
  • syncPolicy = Tự động sync? (prune + selfHeal)

Lưu ý

Application sống ở ns argocd. App thật (Pod) ở ns demo. Application = metadata, không phải app.

Động từNghĩaVí dụ cụ thể
1. SyncĐưa cụm về khớp GitGit thêm Deployment → ArgoCD apply vào cụm
2. Self-Heal
drift remediation		Sửa tay làm lệch Git → kéo về GitGit: 3 pods. kubectl scale 5 → 5 pods. ArgoCD check: "Git nói 3" → scale về 3. Không thể làm liều. Muốn đổi cụm → qua Git.

Lưu ý: Synced ≠ Healthy
ArgoCD có 2 status:
• Synced = Cụm khớp Git? So YAML Git vs cụm. Khớp ✅ / Lệch ❌
• Healthy = Resource chạy tốt? Check Pod status. Running ✅ / Lỗi ❌
VD: Push image: v2 (lỗi) → ArgoCD apply OK → Synced ✅. Pod crash → Degraded ❌

Tại sao cần app-of-apps?

So sánh: Thêm app thủ công vs Tự động

❌ Không dùng app-of-apps

  • Thêm app mới = phải kubectl apply Application thủ công
  • Có 10 app → phải apply 10 lần
  • Không GitOps 100% (Application vẫn apply tay)
  • Quên apply → app không xuất hiện

✅ Dùng app-of-apps

  • Apply root 1 lần duy nhất
  • Thêm app = tạo file YAML + push Git
  • ArgoCD tự phát hiện → tự tạo Application
  • GitOps 100%: Mọi thứ qua Git

Pattern: root Application (chỉ dẫn) → trỏ tới folder chứa nhiều Application con. Root tự tạo con. Giống Deployment tạo ReplicaSet, nhưng level cao hơn.

Rollback

git revert

kubectl rollout undo còn work?

Thí nghiệm tư duy:

# Scenario: 1. Git có image: backend:v2 2. ArgoCD sync → Cụm chạy v2 3. v2 bị lỗi, bạn chạy: kubectl rollout undo deployment/backend 4. Cụm rollback về v1 5. Vài phút sau... điều gì xảy ra? # Đáp án: ArgoCD check: Git nói v2, cụm đang v1 → OutOfSync Self-heal kick in: Apply lại v2 → Cụm lại chạy v2 (version lỗi) → Rollback THẤT BẠI ❌ # Nhớ nguyên tắc số 4: Reconciled. # Cụm phải khớp Git. kubectl rollback # vi phạm nguyên tắc này.

Cách đúng: git revert

# Trong GitOps: Muốn đổi cụm → Đổi Git. # Kể cả rollback. git revert HEAD --no-edit git push # ArgoCD phát hiện commit mới # → Sync cụm về version cũ

So sánh:

kubectl rollout undogit revert
Tác độngChỉ cụmGit + Cụm
Lịch sửKhông (mất sau 10 revisions)Có (Git history vĩnh viễn)
Bị self-heal ghi đè✅ Có❌ Không
Audit trailKhông biết ai rollbackCommit author + message
Thời gianNgay lập tức1-3 phút (ArgoCD poll)

Trade-off: Vài phút latency ↔ safety + traceability.

Lab 5 · app-of-apps

1 "root" quản các app qua 1 thư mục

Thay vì apply tay từng app, để 1 root trông thư mục argocd/apps/ — thêm app sau này chỉ cần thả file + push.

▶ argocd/root.yaml — root trỏ thư mục App con
# file: argocd/root.yaml apiVersion: argoproj.io/v1alpha1 kind: Application metadata: { name: root, namespace: argocd } spec: source: repoURL: https://github.com/<ban>/gitops.git path: argocd/apps # <- thư mục chứa các App con (đã có web.yaml) destination: { server: https://kubernetes.default.svc, namespace: argocd } syncPolicy: { automated: { prune: true, selfHeal: true } }
▶ apply root 1 lần → root tiếp quản web
git add argocd/root.yaml && git commit -m "app-of-apps" git push # apply ROOT bằng tay — LẦN CUỐI dùng kubectl tạo app kubectl apply -f argocd/root.yaml kubectl -n argocd get applications # root + web <- root quản web (đã ở argocd/apps/)
Chốt: từ giờ thêm app mới = thả 1 file vào argocd/apps/ + git push → root tự đẻ, KHÔNG kubectl apply nữa. (Buổi chiều thêm Prometheus/Rollouts/api kiểu này → thấy rõ "1 root nhiều con".)
Xong khi: apply root 1 lần → root + web đều Synced.

Lab 6 · sync waves

Ép thứ tự apply

Đưa Namespace vào Git + thêm ConfigMap (Deployment đọc qua envFrom) → gắn sync-wave để ép thứ tự.

▶ k8s/web.yaml — (3 resource, gắn sync-wave)
# file: k8s/web.yaml kind: ConfigMap # v1, wave 0 metadata: { name: web-config, namespace: demo, annotations: { argocd.argoproj.io/sync-wave: "0" } } data: { MESSAGE: "hello from gitops" } --- kind: Deployment # apps/v1, wave 1 metadata: { name: web, namespace: demo, annotations: { argocd.argoproj.io/sync-wave: "1" } } spec: replicas: 2 selector: { matchLabels: { app: web } } template: metadata: { labels: { app: web } } spec: { containers: [ { name: web, image: nginx:1.27, envFrom: [ { configMapRef: { name: web-config } } ] } ] } --- kind: Service # v1, wave 2 metadata: { name: web, namespace: demo, annotations: { argocd.argoproj.io/sync-wave: "2" } } spec: { selector: { app: web }, ports: [ { port: 80, targetPort: 80 } ] }
▶ k8s/namespace.yaml (wave -1, chạy đầu)
# file: k8s/namespace.yaml apiVersion: v1 kind: Namespace metadata: name: demo annotations: { argocd.argoproj.io/sync-wave: "-1" }
▶ thứ tự wave (nhỏ chạy trước)
Namespace -1 → ConfigMap 0 → Deployment 1 → Service 2
Bẫy nếu thiếu wave: Deployment chạy trước khi ConfigMap tồn tại → CreateContainerConfigError.
Quan sát: push → UI tab Sync → các wave apply đúng thứ tự (Namespace/Config trước Deployment).
